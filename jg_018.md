## 2. 剩余性能卡点

### 2.1 卡点一：CPU 对齐角度循环无批量，rotate_volume 占主导

**现象**：CPU 路径下单颗粒对齐耗时仍以「角度数 × rotate_volume」为主，即使 FFT NCC 已优化平移搜索。

**根因**：

- `core/align.py` 中 `_align_single_scale` 对每个角度三元组调用 `rotate_volume(ref_m, tdrot, tilt, narot)`，使用 `scipy.ndimage.map_coordinates`。
- CPU 路径**无角度批量**；GPU 已有 `_rotate_volume_torch_gpu_batch`，CPU 仍为逐角度 for-loop。
- 典型配置（cone_step=15，inplane_step=15）下，Dynamo 采样可生成数百个角度/颗粒，每角度一次 `map_coordinates`，单颗粒 CPU 耗时仍在 0.5–2 s 量级。

**方案**：

- **CPU 角度预旋转**：对同一 ref，预计算若干角度的 `ref_rot`，缓存后供多颗粒复用（若 seed 相近，角度搜索空间重叠）。
- **Numba/Cython 加速 rotate_volume**：将 `rotate_volume` 热路径用 Numba JIT 或 Cython 重写，减少 Python 与 SciPy 调用开销。
- **粗粒度角度采样**：在文档中建议「初步探索」时使用 cone_step=30、inplane_step=30，收敛后再细化；或增加 `coarse_cone_step`/`coarse_inplane_step` 配置，multigrid 粗阶段使用更粗步长。

---

### 2.2 卡点二：Chunk 内 I/O 与计算完全串行，无预取

**现象**：Chunk worker 内「读颗粒 → 对齐 → 累加 → 丢弃」严格串行，当 I/O 成为瓶颈时（NFS、HDD、网络盘），worker 空闲等待读盘。

**根因**：

- `_alignment_chunk_worker_impl`、`_classification_chunk_worker_impl` 内对 `paths_chunk` 逐项循环：`mrcfile.open` → `align_one_particle` → `local_acc += tr`。
- 无后台预取下一批路径的 MRC，计算与读盘无法重叠。

**方案**：

- **Chunk 内预取**：在 worker 内使用「生产者-消费者」模式：一个线程/协程预取下一颗粒的 MRC 到小队列（如容量 2–4），主循环消费队列做对齐。需控制队列长度避免内存暴增。
- **或主进程预取**：主进程在提交 chunk 前，将 chunk 内路径按「预计访问顺序」排序（如按 rlnImageName 或路径），改善磁盘顺序读；对 NFS 等场景，可考虑将高访问路径的子集预加载到本地 SSD 再跑。

---

### 2.3 卡点三：Classification swap=True 时 N 倍对齐，无跨 ref 并行

**现象**：当 `swap=True` 且多 ref（如 4–8 个）时，每个颗粒需对每个 ref 做一次 `align_one_particle`，总对齐次数 = 颗粒数 × ref 数。

**根因**：

- `_classification_chunk_worker_impl` 内 `for r in refs_to_align` 串行执行，同一颗粒对 N 个 ref 依次对齐。
- 无法在同一颗粒上并行「对齐到 ref A」与「对齐到 ref B」（因需选 best_cc 汇总）。

**方案**：

- **Ref 级并行（进程内）**：将「颗粒 × ref」任务拆为 (颗粒, ref) 对，用 ThreadPoolExecutor 在单进程内并行对齐到不同 ref；对齐完成后主线程汇总每颗粒的 best_ref。注意：`align_one_particle` 内 NumPy/SciPy 会释放 GIL，多线程可并行。
- **或 GPU 上 batch refs**：若使用 GPU，可将「单颗粒 × N ref」扩展为 batch 维，一次 kernel 得到 N 个 CC，取 argmax。
- **swap=False 优化**：当 swap=False 时，每颗粒只对齐到 1 个 ref，无此问题；文档中说明 swap=True 的耗时与 ref 数成正比。

---

### 2.4 卡点四：GPU 路径 shift 仍为逐平移循环

**现象**：GPU 已批量角度旋转，但每个角度下仍对 `shifts` 列表逐项做 `_shift_tensor_zero` + `_ncc_torch`，shift_search=3 时 cube 模式约 7³=343 次。

**根因**：

- `_align_single_scale_torch_gpu` 中 `for sx, sy, sz in shifts` 循环，每次 shift 为整数体素平移，再调用 NCC。
- CPU 路径在 mask 全 1 时可用 `_ncc_volume_fft` 一次性得到所有平移的 NCC 体积；GPU 路径未使用等价 FFT 策略。

**方案**：

- **GPU FFT NCC for shifts**：对给定角度下的 `part_eval_t` 与 `ref_rot_t`，用 `torch.fft` 计算 3D 互相关，一次性得到所有平移的 CC 曲面，再取峰值并做亚像元细化。与 CPU `_ncc_volume_fft` 逻辑一致，需处理 mask 非全 1 时的回退。
- 预期：每角度从 O(|shifts|) 次 shift+NCC 降为 O(1) 次 FFT+argmax，显著减少 kernel 启动与显存带宽。

---

### 2.5 卡点五：颗粒路径顺序未优化，I/O 局部性差

**现象**：当颗粒来自多个 tomogram 且路径在表中交错时，按表顺序读盘会导致随机 I/O，磁盘 seek 增多。

**根因**：

- `paths_list` 通常按 tbl/star 行序，若行序按 tag 或随机，则 `particle_000001.mrc`、`particle_000042.mrc` 可能来自不同 tomogram 目录，读盘跳跃。

**方案**：

- **按路径排序**：在构建 tasks/chunks 前，对 `paths_list` 按 `rlnImageName` 或 `full_path` 排序，使同一目录/同一 tomogram 的颗粒在 chunk 内连续，改善顺序读。
- **按 tomogram 分组后 chunk**：若 tbl 含 tomo 信息，可先按 tomo 分组，组内再按路径排序；chunk 构建时尽量不跨 tomo，减少跨盘访问。需权衡与当前 chunk 构建逻辑的兼容性。

---

### 2.6 卡点六：chunk_size 与 num_workers 缺乏自适应

**现象**：chunk_size 默认 100、num_workers 默认 cpu_count-1，在不同机器、不同 I/O 条件下可能非最优。

**根因**：

- 小 chunk：IPC 次数多，调度开销大。
- 大 chunk：单 chunk 耗时长，负载不均衡加剧；单 worker 内存峰值高（chunk 内需持有多颗粒的中间结果，实际已流式处理，主要持 local_acc）。
- num_workers 过大：在 I/O 瓶颈时，多进程争抢磁盘带宽，反而降速。

**方案**：

- **文档化调优建议**：在 README 或配置说明中增加「60k 颗粒推荐配置」表，给出 chunk_size、num_workers 的典型范围及「I/O 瓶颈时适当减少 num_workers」的说明。
- **可选 auto 模式**：`chunk_size: 0` 或 `auto` 时，根据 `len(paths)` 与 `num_workers` 自动选择，如 `chunk_size = max(50, min(200, len(paths) // (num_workers * 4)))`，使 chunk 数约为 workers 的 4–8 倍，兼顾负载均衡与 IPC。

---

### 2.7 卡点七：Reconstruction chunk 内无预取

**现象**：Reconstruction 的 `_reconstruction_chunk_worker` 内串行读颗粒、变换、累加，与 alignment/classification 类似。

**根因**：

- 单颗粒 reconstruction 本身较轻（transform + 累加），但 60k 量级时 I/O 仍可能成为瓶颈。
- 当前无预取，读与算串行。

**方案**：

- 与 2.2 类似：chunk 内预取下一颗粒的 MRC，使读与算重叠。Reconstruction 单颗粒更轻，预取收益可能小于 alignment，可列为 P2，在 I/O 实测成为瓶颈时再实现。

---

### 2.8 卡点八：tbl 流式在 chunk 模式下被禁用

**现象**：Chunk 模式下主进程收集所有 rows 后排序再写 tbl，无法边算边写。

**根因**：

- p_016 设计：chunk 返回的 rows 乱序（按 chunk 完成顺序），需收集后按 index 排序再写。
- 对 60k，主进程持 60k 个 row dict，内存约数十 MB，尚可接受；若规模更大（如 200k+），可能成为压力。

**方案**：

- **短期**：保持现状，在文档中说明 chunk 模式下 tbl 非流式，大规模时注意内存。
- **中期**：若需支持超大规模，可考虑「chunk 内按 index 排序后返回」「主进程按 chunk 顺序合并」或「两阶段：先写临时文件，再 merge 排序」，避免主进程一次性持全部 rows。

## 3. 解决方案优先级与预期收益

| 优先级 | 卡点 | 方案摘要 | 预期收益 |
|--------|------|----------|----------|
| **P0** | 2.1 CPU 角度无批量 | Numba 加速 rotate_volume 或粗粒度采样建议 | 单颗粒 CPU 时间可降 20–50%（视实现） |
| **P0** | 2.4 GPU shift 逐平移 | GPU FFT NCC for shifts，一次性得平移 CC | GPU 路径每角度从 O(shifts) 降为 O(1) |
| **P1** | 2.2 Chunk 内无预取 | Worker 内预取队列，读与算重叠 | I/O 瓶颈时吞吐提升 10–30% |
| **P1** | 2.3 Classification swap N 倍 | Ref 级并行或 GPU batch refs | swap=True 且多 ref 时显著加速 |
| **P1** | 2.5 路径顺序 | 按路径/tomo 排序后再 chunk | 改善 I/O 局部性，慢盘时收益明显 |
| **P2** | 2.6 chunk_size 自适应 | 文档化 + 可选 auto | 配置合理化，减少无效调优 |
| **P2** | 2.7 Reconstruction 预取 | 同 2.2 | I/O 瓶颈时略提升 |
| **P2** | 2.8 tbl 流式 | 文档化；超大规模时两阶段写 | 当前 60k 可接受 |

## 4. 实施路线图（仅规划）

### 阶段一：高优先级（P0）

1. **CPU rotate_volume 加速**  
   - 方案 A：用 Numba `@njit` 重写 `rotate_volume` 内层（坐标变换 + 插值），与 SciPy `map_coordinates` 结果对比验证。  
   - 方案 B：在配置/文档中增加「粗粒度探索」建议（cone_step=30, inplane_step=30），并评估 multigrid coarse 阶段步长放大。

2. **GPU FFT NCC for shifts**  
   - 在 `_align_single_scale_torch_gpu` 中，当 mask 全 1 时，用 `torch.fft.fftn` / `ifftn` 计算 3D 互相关，替代 `for sx,sy,sz in shifts` 循环。  
   - 取 argmax 得整数平移，再亚像元细化（可复用现有 subpixel 逻辑）。

### 阶段二：中优先级（P1）

3. **Chunk 内预取**  
   - 在 `_alignment_chunk_worker_impl`、`_classification_chunk_worker_impl` 内增加预取线程/队列，队列容量可配置（如 2–4）。

4. **Classification ref 级并行**  
   - 当 swap=True 且 nref>1 时，在 chunk worker 内用 ThreadPoolExecutor 并行「对齐到 ref r」，主线程汇总 best_ref。

5. **路径排序**  
   - 在 alignment/classification 构建 tasks 前，对 paths 按 `full_path` 或 `rlnImageName` 排序。

### 阶段三：低优先级（P2）

6. **chunk_size auto**  
   - 实现 `chunk_size: 0` 或 `auto` 时的自适应公式，并文档化。

7. **Reconstruction 预取**  
   - 与 2.2 类似，在 reconstruction chunk worker 内增加预取。

8. **tbl 流式与大规模**  
   - 文档化 chunk 模式下 tbl 行为；若后续支持 200k+ 规模，再设计两阶段写。
