## 2. 60,000 颗粒规模下的性能评估与比较

### 2.1 假设与量级

- **颗粒数**：60,000  
- **Box 尺寸**：以 48³ 与 64³ 为典型（约 0.11M 与 0.26M voxel/颗粒）  
- **单颗粒对齐**（CPU，cone_step=15，inplane_step=15，shift_search=3，multigrid_levels=2）：量级约 0.5–2 s/颗粒（视 box 与是否用 FFT NCC、是否 full mask）  
- **单颗粒 reconstruction**（transform + 累加）：量级约 1–10 ms/颗粒（远轻于对齐）

以下为**量级估算**，用于对比与规划，非精确基准。

### 2.2 Alignment：单进程 vs 多进程（60k）

| 配置 | 单颗粒耗时假设 | 60k 总耗时（量级） | 说明 |
|------|----------------|---------------------|------|
| num_workers=1 | 1 s | ~16.7 h | 单核跑满，为基线 |
| num_workers=8 | 1 s（并行） | ~2.1 h | 理想线性加速；实际有进程调度与 ref 加载 |
| num_workers=16 | 1 s（并行） | ~1.0 h | 同上；超过物理核数收益递减 |
| num_workers=0（auto） | — | 取决于 cpu_count | 建议 60k 时显式设 8–16，避免占满机器 |

**当前瓶颈**：  
- 每个 CPU worker 独立从磁盘加载 reference（及 mask）；多进程时 ref 被加载 N_workers 次，内存总占用 ≈ N_workers × ref_size。  
- 60k 个任务即 60k 次 `submit`，进程池调度与结果回收有开销；单次任务若过轻，IPC 占比会上升。

### 2.3 Reconstruction：单进程 vs 多进程（60k）

| 配置 | 单颗粒耗时假设 | 60k 总耗时（量级） | 说明 |
|------|----------------|---------------------|------|
| recon_workers=1 | 5 ms | ~5 min | 以 transform+sum 为主，I/O 随存储变化 |
| recon_workers=8 | 5 ms（并行） | ~40 s | 按块分配，主进程合并 fft_sum/acc |
| recon_workers=16 | 5 ms（并行） | ~20 s | 块更小，单块约 3750 颗粒 |

**内存注意**：  
- 每块内 worker 需读入 chunk 内所有颗粒并做 transform+累加；chunk_size = ceil(60000 / recon_workers)。  
- 例如 16 workers、box=64：单块约 3750 颗粒，若逐颗粒读-算-丢则峰值约 1× box³ × 4B；若实现为一次读入多颗粒再算，则需控制 chunk_size 或分批，避免单进程内存过大（如 3750×64³×4B ≈ 3.9 GB/worker）。

### 2.4 Classification（MRA，60k × 每轮）

| 项目 | 修改前（jg_012 前） | 修改后（当前） |
|------|----------------------|----------------|
| 每轮迭代粒子读盘次数 | 2×（对齐 1 次 + 平均按 ref 再读 1 次） | 1×（对齐时即累加 tr，平均阶段不重读） |
| 60k 单轮 I/O 量（估算） | ~120k 次 MRC open | ~60k 次 MRC open |
| 多进程 | 无 | num_workers>1 时 ProcessPoolExecutor，worker 内缓存 ref/mask |

**结论**：同一迭代内 I/O 约减半；多轮迭代 × 多 ref 时，总耗时仍以对齐为主，多进程 + 流式累加已显著改善。

### 2.5 Crop（60k 颗粒）

- 已按 tomogram 分组，组内 ThreadPoolExecutor；多 tomogram 时多组并行。  
- 60k 颗粒若来自少量大 tomogram，则组内线程并行有效；若来自很多小 tomogram，则并行度受组数限制。  
- 写盘：60k 个小 MRC 文件，可能受文件系统小文件写入性能限制；当前无异步写队列。

## 3. 仍可提升的性能点（60k 规模）

### 3.1 Alignment / Classification

| 方向 | 说明 | 预期收益 |
|------|------|----------|
| **Ref/Mask 共享，减少每 worker 加载** | 当前每个 CPU worker 自行从磁盘读 reference（及 mask）。可考虑：主进程读 ref/mask 后以 initializer + 全局变量注入 worker（类似 benchmark_profiles 的 `_align_worker_init`），或使用 shared memory / 只读 mmap，使 N 个 worker 不重复加载 N 份。 | 降内存、减 I/O，缩短启动与冷缓存阶段 |
| **按 chunk 提交任务** | 将 60k 粒子打成 chunk（如每 chunk 50–200 颗粒），每任务处理一整块后返回 (rows, local_avg_acc)。主进程合并表格与 avg_acc。可减少 submit/result 次数与序列化开销。 | 降低进程池与 IPC 开销，尤其在高 worker 数时 |
| **带 mask 时的 FFT NCC** | 当前 `_ncc_volume_fft` 仅在 mask 全 1 时使用；若用户使用 nmask（非全 1），则回退到逐平移实空间 NCC。可探索 mask 下的 FFT NCC 近似或分块 FFT，在保持精度的前提下减少计算。 | 使用 nmask 时单颗粒对齐时间明显下降 |
| **GPU 批量** | GPU 路径仍为单粒子、单角度循环。可做「多角度/多粒子」批量 kernel，减少 kernel 启动与 CPU–GPU 往返。 | GPU 利用率与吞吐提升 |

### 3.2 Reconstruction

| 方向 | 说明 | 预期收益 |
|------|------|----------|
| **Chunk 内流式处理** | 当前 chunk worker 可能先读入整块路径列表再逐颗粒读-变换-累加。应保证 chunk 内是「读一个、算一个、累加、丢弃」，不在内存中保留整块颗粒，以控制峰值内存。 | 避免 60k/16 时单 worker 数 GB 级内存 |
| **recon_workers 建议值** | 60k 时 recon_workers=8 或 16 即可接近 I/O 或单机带宽上限；过大反而增加调度与合并成本。在文档中给出「60k 推荐 recon_workers」可避免用户盲目开大。 | 配置合理化，减少无效多进程开销 |

### 3.3 I/O 与内存

| 方向 | 说明 | 预期收益 |
|------|------|----------|
| **MRC 只读 mmap 常态化** | 当前已用 `np.asarray(mrc.data)` 且按需 copy；若 mrcfile 在 `mode="r"` 下提供的是 mmap，则多进程下同文件可共享只读页。确认并统一「只读路径」不触发不必要的 copy。 | 多 worker 时总内存与读盘进一步降低 |
| **预取** | 在单线程或单 worker 内，用后台线程/队列预取下一批粒子路径的 MRC，使计算与读盘重叠。需限制队列长度。 | 在 I/O 成为瓶颈时提升吞吐 |

### 3.4 Crop

| 方向 | 说明 | 预期收益 |
|------|------|----------|
| **异步写队列** | 将 crop 结果写入队列，由专用线程负责写 MRC 文件，避免 worker 被慢盘阻塞。 | 高 I/O 负载时提高 CPU 利用率 |
| **num_workers 与 tomogram 数** | 当 tomogram 数量远小于 num_workers 时，部分 worker 闲置。可文档说明：建议 num_workers 不超过「tomogram 数 × 每 tomo 期望并行数」或 CPU 核数。 | 配置更清晰 |

## 4. 60,000 颗粒推荐配置（规划）

以下为**规划建议**，便于用户与运维在 60k 规模下获得可接受的性能与稳定性。

| 命令 | 参数 | 推荐范围/值 | 备注 |
|------|------|-------------|------|
| alignment | num_workers | 8–16 或 0（auto） | 视 CPU 核数；60k 建议显式 8–16 |
| alignment | multigrid_levels | 2 | 保持 coarse-to-fine，控制单颗粒时间 |
| alignment | cone_step / inplane_step | 15 | 更细步长会显著增加时间 |
| alignment | device | cuda 若可用 | GPU 可大幅缩短 wall time，需后续做批量 kernel |
| reconstruction | recon_workers | 8–16 | 与 alignment 类似；过大会增加内存与调度 |
| classification | num_workers | 8–16 | 与 alignment 一致 |
| classification | max_iterations | 3–5 | 按需；每轮已只读 1 次粒子 |
| crop | num_workers | 4–16 | 受 tomogram 数与 I/O 限制 |

**内存粗算（60k，box=64）**：  
- 单 ref：64³×4B ≈ 1 MB。  
- 单颗粒：1 MB。N 个 worker 各持 1 ref + 1 颗粒时 ≈ N×(1+1) MB；若 ref 共享则约 N×1 MB + 1 MB。  
- Reconstruction 每 worker 块内流式处理时，峰值约 1–2 个颗粒体积；若 16 workers 则主进程还持 16 个 chunk 的累加器（每个 1×64³×8B ≈ 2 MB），约 32 MB，可接受。

## 5. 性能评估对比小结（60k）

| 场景 | 基线（单进程/修改前） | 当前实现（多进程+流式+FFT NCC 等） | 若再做 3.1–3.4 优化 |
|------|------------------------|--------------------------------------|----------------------|
| Alignment 60k | ~16.7 h（1 s/颗粒） | ~1–2 h（8–16 workers） | 进一步降 ref 加载与 IPC；mask 下 FFT 可再降单颗粒时间 |
| Reconstruction 60k | ~5 min | ~0.5–1 min（8–16 workers） | 保证 chunk 流式、推荐 recon_workers 即可 |
| Classification 60k/轮 | 2× 读盘 + 单线程 | 1× 读盘 + 多进程 + 流式累加 | ref 共享、chunk 提交可再降内存与调度 |
| Crop 60k | 依赖 tomogram 分组与线程数 | 同左，可配置 num_workers | 异步写、num_workers 建议可减少 I/O 阻塞 |

## 6. 建议的后续步骤（仅规划）

1. **文档与默认值**：在 README 或配置说明中增加「60k 颗粒规模」的推荐配置表（如本节 4 表），以及 `num_workers`/`recon_workers` 的取值说明。  
2. **Benchmark 扩展**：在 `benchmark_profiles` 中增加「60k 量级」的 profile（如 1000 颗粒 × num_workers=4/8），或单独脚本用 60k 子集做端到端计时，用于回归与容量规划。  
3. **实现优先级**：  
   - 高：Ref/mask 在 CPU 多进程下共享（initializer 或 shared memory）、Reconstruction chunk 内严格流式。  
   - 中：Alignment/Classification 按 chunk 提交；带 mask 的 FFT NCC 探索。  
   - 低：Crop 异步写、I/O 预取、GPU 批量 kernel。

以上为性能评估与 60,000 颗粒规模下的优化规划，输出至 **jg_0015.md**。
