# pydynamo 性能优化后续方案（jg_020）

本文档基于 jg_019.md 规划、p_019.md 实施、r_019.md 进度及 f_019.md 审查反馈，在**第一轮行动循环完成后**，比较规划前后变化，并评估是否还有进一步性能提升空间。**仅输出方案，不实现代码**。

| jg_019 卡点 | 实施状态 | 预期收益 |
|-------------|----------|----------|

## 2. 规划前后性能变化（定性）

- **GPU 对齐**：网格与 NCC 均在 GPU 上批量完成，减少 CPU 参与与 kernel 启动次数。
- **Classification GPU**：swap 时多 ref 并行，预取与 I/O 重叠。
- **预期**：在 60k 颗粒、全 mask、多 ref 场景下，GPU 路径整体可望有 20–50% 的 wall time 改善（视硬件与 I/O 情况而定）。

---

## 3. 剩余可优化空间

### 3.1 颗粒级 GPU 批处理（未实施）

**现象**：当前仍为「单颗粒 → 多角度 batch」模式，颗粒之间串行。多颗粒可组成 batch 维，一次 kernel 处理多颗粒 × 多角度。

**方案**：在 `align_one_particle` 或调用方，支持 `particles [P, D, H, W]` 输入，内部扩展为 `[P, B, D, H, W]` 与 ref batch 做 NCC，得到 `[P, B]` 的 best CC，再取全局 argmax。实现复杂，需重构接口与多 GPU 调度。

**优先级**：P2，在单颗粒优化饱和后再考虑。

---

### 3.2 非全 mask 下的 GPU FFT NCC 近似（jg_018 遗留）

**现象**：`nmask` 非全 1 时回退到实空间 NCC，GPU 路径仍对每个 shift 做 `_shift_tensor_zero` + `_ncc_torch`，耗时长。

**方案**：探索 mask 内 FFT NCC 近似（mask 外置零），或分块 FFT，与 Dynamo 行为对比验证。需保证与现有 `cc_mode`、`wedge` 一致。

**优先级**：P2，使用 nmask 时收益大。

---

### 3.3 CPU 路径 rotate_volume 加速（jg_018 遗留）

**现象**：CPU 路径仍对每个角度调用 `scipy.ndimage.map_coordinates`，无批量。

**方案**：Numba `@njit` 重写 `rotate_volume` 内层，或文档化粗粒度采样建议（cone_step=30, inplane_step=30）。

**优先级**：P2。

---

### 3.4 多进程与 GPU 共存（jg_019 卡点 2.5）

**现象**：`num_workers > 1` 时 classification 强制 CPU，无法同时利用多进程与 GPU。

**方案**：探索 `spawn` 启动 ProcessPoolExecutor，每 worker 绑定单 GPU 并独立初始化 CUDA。

**优先级**：P2，实现与测试成本较高。

---

## 4. 结论

- 本轮实施的 P0/P1 项已完成，审查反馈已修复。
- 当前主要瓶颈已缓解，剩余优化多为 P2，收益与实现成本需按实际场景权衡。
- **建议**：在实际 60k 颗粒规模下做一次基准测试，量化 GPU 路径的 wall time 改善，再决定是否推进 3.1–3.4。
