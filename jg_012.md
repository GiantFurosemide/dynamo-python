# pydynamo 性能卡点分析与解决方案规划

## 1. 现状与结论

基于对 pydynamo 代码库的浏览，主要耗时路径集中在：**alignment**、**classification**、**reconstruction**、**crop**。下面按「卡点 → 根因 → 方案」给出结论与规划，**仅输出方案，不实现代码**。规划文档输出到 **jg_012.md**。

---

## 2. 卡点一：Alignment / Classification 在 CPU 上单线程执行

**现象**：多核机器上跑 alignment 或 classification 时，仅单核满载，其余核心闲置。

**根因**：

- [commands/alignment.py](pydynamo/src/pydynamo/commands/alignment.py) 与 [commands/classification.py](pydynamo/src/pydynamo/commands/classification.py) 在 CPU 路径下对粒子循环是**单线程 for-loop**（约 alignment 291–318 行、classification 295–320 行）。
- 仅当 `device == "cuda"` 且 `len(gpu_ids) > 1` 时才使用 `ThreadPoolExecutor` 做多 GPU 调度；CPU 时从不使用多进程/多线程处理粒子。
- Python GIL 下，NumPy/SciPy 的 CPU 计算仍可释放 GIL，但当前「每粒子：读 MRC + align_one_particle」的整体任务是在主线程串行执行，没有把「粒子级任务」分给多进程，因此多核无法被利用。

**方案**：

- 引入 **进程级并行**：用 `multiprocessing.Pool` 或 `concurrent.futures.ProcessPoolExecutor`，按粒子（或按 chunk）划分任务，每个 worker 独立执行「读 MRC + align_one_particle」并返回结果；主进程汇总表格与 average。
- 配置项：在 alignment/classification 的 YAML 中增加 `num_workers`（或 `cpu_workers`），表示 CPU 路径下用于处理粒子的进程数；默认可与 `os.cpu_count()` 或略减一，避免占满机器。
- 注意：多进程下 reference/mask 等只读数据需在每个 worker 内加载或通过 fork 继承，避免大对象重复序列化；可考虑「主进程加载 ref/mask → fork 后子进程只读」，或每个 worker 读一次并做简单缓存（按 config path 或 inode）。

## 3. 卡点二：Alignment 算法为「逐角度 + 逐平移」实空间 NCC

**现象**：单粒子对齐耗时随角度/平移搜索规模与 box 尺寸快速增长，成为整体耗时的主要部分。

**根因**：

- [core/align.py](pydynamo/src/pydynamo/core/align.py) 中单尺度对齐 `_align_single_scale` 对每个角度三元组 `(tdrot, tilt, narot)` 做：`rotate_volume`（SciPy `map_coordinates`）→ `shift` → `normalized_cross_correlation` 或 `_local_normalized_cross_correlation`，均为实空间操作。
- 未使用「FFT 一次性得到平移空间 + 多角度预计算/插值」等批量策略；角度数与平移数相乘再乘体素规模，导致计算量巨大。
- 默认配置如 `cone_step=15`、`inplane_step=15`、`shift_search=3` 会生成大量候选，单粒子耗时在 CPU 上很容易到秒级。

**方案**：

- **短期**：在保持当前 API 与精度的前提下，做算法层优化：
  - **FFT-based NCC for shifts**：对给定角度下的 reference 与 particle，用 FFT 计算整幅 3D 的平移 NCC 曲面，再取峰值并做亚像元细化，避免对每个平移候选逐次 shift + NCC。
  - **角度采样与多尺度**：已有 multigrid；可再评估 coarse 阶段是否可进一步放大步长或减少 tilt/inplane 点数，在保证收敛前提下减少 fine 阶段候选数。
- **中期**：考虑 **参考体积预旋转缓存**（或预计算若干角度的 FFT），减少每粒子重复的 `rotate_volume` 调用；或对 GPU 路径做 **批量角度/批量粒子** 的 kernel，提高 GPU 利用率。
- 上述改动需与现有 `wedge`、`mask`、`subpixel` 等行为一致，并保留 regression 测试（如 [scripts/benchmark_profiles.py](pydynamo/src/pydynamo/scripts/benchmark_profiles.py)）。

## 4. 卡点三：Reconstruction 单线程且逐粒子读盘

**现象**：Reconstruction 阶段整体偏慢，且 CPU 只用到单核。

**根因**：

- [commands/reconstruction.py](pydynamo/src/pydynamo/commands/reconstruction.py) 中 realspace 与 Fourier 两条路径均为单线程：对 `paths_list` 逐粒子循环，每次 `mrcfile.open(...).data.copy()`，再 `apply_inverse_transform` 并累加（约 197–237 行、258 行附近）。
- 无进程/线程级并行，也无批量读或 prefetch。

**方案**：

- 引入 **进程池或线程 + 预取**：
  - 用 `ProcessPoolExecutor` 按粒子块（chunk）分配：每个任务为「一批粒子路径 + 角度/平移表」，返回局部累加结果；主进程合并各块 sum 与计数。
  - 或使用「生产者-消费者」：一个线程/进程专门读 MRC 到队列，多个 worker 做 `apply_inverse_transform` + 累加，降低 I/O 与计算耦合。
- 若单次 reconstruction 粒子数极大，可增加配置项 `recon_workers`，默认 1 保持现状，大于 1 时启用上述并行。

## 5. 卡点四：Classification 每轮迭代中粒子被重复读盘

**现象**：MRA 多轮迭代时，每轮除「对齐阶段读一次粒子」外，在「按 ref 做平均」时再次按粒子路径逐个打开 MRC（见 [commands/classification.py](pydynamo/src/pydynamo/commands/classification.py) 约 325–350 行），同一粒子在同一迭代内被读多次。

**根因**：

- 对齐阶段：每个粒子读一次，对齐到多个 ref，得到每粒子最佳 ref 与 pose。
- 平均阶段：按 ref 分类后，对每个 ref 的 JSONL 行再遍历，按 `rlnImageName` 打开粒子文件，做 `apply_inverse_transform` 并累加。因此同一粒子若只属于一个 ref，也会在对齐时读 1 次、在平均时再读 1 次；若实现上是「按 ref 循环再按粒子循环」，则每个粒子被读次数 = 其所属 ref 的参与平均的 ref 数（当前实现是每个 ref 独立循环粒子，所以每个粒子在平均阶段只再读 1 次，但对齐+平均共 2 次）。

**方案**：

- **流式对齐 + 按 ref 累积**：在对齐阶段，每完成一个粒子就根据其最佳 ref 与 pose，把 `apply_inverse_transform` 的结果累加到对应 ref 的 buffer，然后丢弃该粒子内存；平均阶段不再按粒子路径重新读 MRC，只对各 ref buffer 做归一化与对称性等后处理。这样每粒子整轮迭代只读 1 次。
- 若希望保持「先对齐、后平均」两阶段结构，可改为：对齐阶段输出 (particle_path, ref_id, pose)；平均阶段**按粒子路径排序/去重**，对每个路径只读一次，然后根据 (path → ref_id, pose) 映射把变换结果累加到对应 ref。避免同一 path 被多次 open。

## 6. 卡点五：粒子 I/O 全量拷贝、无 mmap/预取

**现象**：大量粒子时，`mrcfile.open(..., mode="r").data.copy()` 导致内存与 I/O 峰值高，且可能阻塞计算。

**根因**：

- Alignment/Classification/Reconstruction 中粒子一律通过 `mrcfile.open(..., permissive=True)` 打开后 `.data.copy()` 或 `.astype(np.float32)` 拷贝（见 grep 结果与各 command 文件），没有使用只读 mmap 或分块读。
- 拷贝既增加内存占用，也增加总线与内存带宽压力；在单线程下读与算串行，在多进程下若每个 worker 都拷贝，总内存会成倍增加。

**方案**：

- **只读 mmap**：对单文件 MRC，在 `mrcfile` 支持的前提下使用 `mode="r"` 且不 `.copy()`，用 `np.asarray(mrc.data)` 或等价方式得到只读视图，在需要写或持久化时再在局部做拷贝。这样多进程 fork 后同一文件可共享只读页，减少物理内存与读盘。
- **可选预取**：在单线程或 worker 内，对「下一批粒子路径」做异步或后台读入（例如用线程队列），使计算与 I/O 重叠；需要控制队列长度以限制内存。
- 若使用 ProcessPoolExecutor，每个 worker 内按路径打开时优先 mmap，避免主进程一次性把全部粒子读入再分发。

## 7. 卡点六：Crop 的并行与 I/O

**现象**：Crop 在大体量时可能受 I/O 或单 tomogram 内并行度限制。

**根因**：

- [commands/crop.py](pydynamo/src/pydynamo/commands/crop.py) 已按 tomogram 分组，同一 tomogram 内用 `ThreadPoolExecutor` 并行 crop；多 tomogram 时再用线程池并行处理不同组（约 264–286 行）。
- 每个 crop 任务里会 `save_subtomo` 写单文件，若磁盘慢或文件系统小文件写入效率低，会形成瓶颈；且当前并行单位是「tomogram 组」或「组内粒子」，没有进程级 CPU 并行，crop 本身是 CPU 轻量（主要是 slice + 写），可能更多是 I/O bound。

**方案**：

- **保持现有线程并行**，先做可配置：如 `num_workers` 控制每 tomogram 内线程数及 tomogram 组并行数，避免过度订阅 I/O。
- 若实测为 I/O 瓶颈：可考虑 **批量写**（例如先写入内存 buffer 或临时目录，再批量 flush）或 **异步写队列**（专门线程负责写，worker 只做 crop 与入队），减少阻塞。
- 若 crop 的 slice 与 mask 等计算在将来变重，可再考虑对「组内粒子」使用 ProcessPoolExecutor，与当前「按组并行」结合。

---

## 8. 其他可选项（不展开实现）

- **GPU 路径**：当前 GPU 单粒子单角度循环，GPU 利用率可能不高；后续可考虑批量角度或批量粒子 kernel，减少 CPU–GPU 往返与 kernel 启动次数。
- **配置与可观测性**：在 YAML 中显式暴露 `num_workers` / `recon_workers` / `crop_workers` 等，便于调优；已有 `progress_log_every` 与 RSS 等，可增加「当前 worker 数」「估计剩余时间」等，便于定位卡点。
- **Benchmark**：[scripts/benchmark_profiles.py](pydynamo/src/pydynamo/scripts/benchmark_profiles.py) 仅测单进程 alignment 吞吐；可增加「多进程 alignment」「reconstruction 多 worker」的 profile，用于回归与容量规划。

---

## 9. 实施优先级建议（仅规划）

| 优先级 | 卡点 | 方案摘要 | 预期收益 |
|--------|------|----------|----------|
| P0 | Alignment/Classification CPU 单线程 | 引入 ProcessPoolExecutor + num_workers | 多核线性加速，收益最大 |
| P0 | Classification 重复读盘 | 对齐时即累加到 ref buffer，或平均阶段按 path 去重只读一次 | 每轮迭代 I/O 减半或更多 |
| P1 | Alignment 实空间 NCC | FFT-based 平移 NCC + 可选角度缓存/批量 | 单粒子对齐时间显著下降 |
| P1 | Reconstruction 单线程 | ProcessPoolExecutor 按块累加 | 多核加速 |
| P2 | 粒子 I/O 全量 copy | mmap 只读 + 可选预取 | 降内存、略重叠 I/O/计算 |
| P2 | Crop | 可配置 workers + 可选异步写 | 避免 I/O 阻塞与过度订阅 |

以上方案均以「先提方案、不实现」为前提；具体实现时需兼顾现有测试、配置与 Dynamo/RELION 兼容性。
