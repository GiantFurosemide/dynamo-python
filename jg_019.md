# pydynamo 性能卡点分析与解决方案规划（jg_019）

本文档基于对 pydynamo 源码的实际浏览及 jg_016、jg_018 的既有规划，针对**实际运行性能仍达不到使用需求**的问题，聚焦 **align 与 classification 的 GPU 部分**，识别剩余卡点并提出解决方案。**仅输出方案，不实现代码**。

## 2. 剩余性能卡点（聚焦 align/classification GPU）

### 2.1 卡点一：GPU 旋转网格在 CPU 上计算，存在 CPU-GPU 传输

**现象**：`_rotate_volume_torch_gpu_batch` 对每个角度在 NumPy 上计算旋转网格（`np.mgrid`、`Rotation.from_euler`、矩阵乘法），再 `np.stack` 后一次性 `torch.as_tensor` 传到 GPU。

**根因**：
- `core/align.py` 第 794–813 行：对 `batch_triplets` 中每个 `(tdrot, tilt, narot)` 在 CPU 上计算 `grid_np`，循环内大量 NumPy 操作。
- 批量 32 角度时，32 次 `Rotation.from_euler`、32 次 `np.mgrid`、32 次矩阵乘法，最后一次性传输 `[B, D, H, W, 3]` 到 GPU。
- **影响**：CPU 计算与 GPU 传输串行，GPU 在等待；大体积（如 64³）时单批网格约 32×64³×3×4B ≈ 100 MB 传输。

**方案**：
- **GPU 端网格计算**：用 PyTorch 在 GPU 上生成 `grid_batch`，避免 CPU 循环与传输。对 ZXZ 欧拉角，可用 `torch.einsum` 或 `torch.bmm` 实现旋转矩阵乘法。
- **预期**：消除 CPU 网格计算与传输，减少 kernel 启动前的等待。

---

### 2.2 卡点二：GPU 路径 NCC 按角度逐次计算，未批量

**现象**：`_align_single_scale_torch_gpu` 中，`ref_rot_batch` 已批量旋转得到 `[B, D, H, W]`，但 NCC 在 `for i_angle, (tdrot, tilt, narot) in enumerate(batch_triplets)` 内逐角度调用 `_ncc_volume_fft_torch(part_eval_t, ref_t)`。

**根因**：
- `_ncc_volume_fft_torch` 接收单个体积，内部 `torch.fft.fftn` 等为单体积接口。
- 批量旋转后仍按角度循环，每角度一次 FFT + IFFT + argmax，无法利用批量 FFT。
- **影响**：GPU 利用率不足，batch 维度未充分利用。

**方案**：
- **批量 FFT NCC**：扩展 `_ncc_volume_fft_torch` 为 `_ncc_volume_fft_torch_batch(part_eval_t, ref_rot_batch)`，`part_eval_t` 扩展为 `[B, D, H, W]`（复制 B 份）或保持单体积与 `ref_rot_batch [B, D, H, W]` 逐对计算，用 `torch.fft.fftn(..., dim=(-3,-2,-1))` 对 batch 做批量 FFT。
- 一次性得到 `[B, D, H, W]` 的 NCC 体积，再对每个 b 在 `shift_search_mask_t` 内取 argmax，得到 B 个最佳平移。
- **预期**：每批从 B 次 FFT 降为 1 次批量 FFT，显著减少 kernel 启动与显存带宽。

---

### 2.3 卡点三：Classification GPU 路径 swap 时 ref 串行

**现象**：当 `device=cuda` 且 `swap=True` 时，`_align_particle_task` 内 `for r in refs_to_align` 串行执行，同一颗粒对 N 个 ref 依次调用 `align_one_particle`。

**根因**：
- `classification.py` 第 706–742 行：GPU 路径（ThreadPoolExecutor 调度颗粒）下，每颗粒内部 `for r in refs_to_align` 无并行。
- CPU chunk 路径已有 `ThreadPoolExecutor` 并行多 ref（第 429–465 行），GPU 路径未做等价优化。
- **影响**：swap=True 且 nref=4–8 时，单颗粒 GPU 时间 ≈ nref × 单次对齐时间。

**方案**：
- **GPU 路径 ref 级并行**：在 `_align_particle_task` 内，当 `swap and nref > 1` 且 `resolved_device == "cuda"` 时，用 `ThreadPoolExecutor` 并行执行 `align_one_particle(..., device="cuda", device_id=...)` 到不同 ref，主线程汇总 best_ref。多 ref 对齐可分布到不同 GPU（若 `len(gpu_ids) >= nref`）或同一 GPU 多线程（GIL 在 PyTorch 运算时释放）。
- **或 GPU batch refs**：将「单颗粒 × N ref」扩展为 batch 维，一次 `align_one_particle` 支持多 ref 输入，在 core/align 内批量计算 N 个 CC 并取 argmax。实现复杂度较高，可作为后续优化。

---

### 2.4 卡点四：Alignment/Classification GPU 路径无 I/O 预取

**现象**：GPU 路径下，主进程对每个颗粒执行：`mrcfile.open` → `np.asarray` → 上传 GPU → `align_one_particle`，读盘与计算完全串行。

**根因**：
- Alignment GPU：`_run_one` 内直接 `mrcfile.open` 读颗粒，然后 `align_one_particle`，无预取队列。
- Classification GPU：`_align_particle_task` 内同样直接读颗粒再对齐。
- Chunk 模式的 `prefetch_queue_size` 仅用于 CPU 多进程 chunk worker，GPU 路径未使用。
- **影响**：I/O 瓶颈（NFS、HDD）时，GPU 空闲等待读盘。

**方案**：
- **GPU 路径预取**：在主进程内增加预取线程/队列，后台预加载下一批颗粒（如 2–4 个）到内存，主循环消费队列做对齐。队列满时阻塞提交，避免内存暴增。
- 与 CPU chunk 预取类似，但作用域为主进程的 GPU 单/多线程路径。
- **预期**：I/O 瓶颈时吞吐提升 10–30%。

---

### 2.5 卡点五：Classification 多进程与 GPU 互斥

**现象**：当 `num_workers > 1` 时，classification 使用 ProcessPoolExecutor + chunk worker，chunk worker 内 `align_one_particle(..., device="cpu")` 固定为 CPU。用户无法同时获得「多进程 + GPU」。

**根因**：
- 多进程 fork 后子进程与主进程 GPU 上下文不共享，且 CUDA 与 fork 存在兼容性问题。
- 当前设计：CPU 多进程（chunk）或 GPU 单/多线程（ThreadPoolExecutor 按颗粒分配 GPU），二者二选一。
- **影响**：大规模颗粒时，用户若选 GPU 则无法利用多进程 I/O 与负载均衡；若选多进程则无法用 GPU 加速对齐。

**方案**：
- **短期**：文档化「num_workers>1 时强制 CPU」行为，建议 I/O 瓶颈时用 CPU 多进程，计算瓶颈时用 GPU 单线程并配合预取。
- **中期**：探索「多进程 + 每进程单 GPU」：每 worker 绑定固定 GPU，worker 内使用 `device_id` 执行 GPU 对齐。需解决 fork 后 CUDA 初始化（如 `torch.cuda.init()` 或 spawn 替代 fork）。
- **长期**：若支持 spawn 启动 ProcessPoolExecutor，可让每个 worker 独立初始化 CUDA，实现多进程 × 多 GPU。

---

### 2.6 卡点六：CPU 路径 rotate_volume 无角度批量（jg_018 遗留）

**现象**：CPU 路径下，`_align_single_scale` 对每个角度调用 `rotate_volume(ref_m, tdrot, tilt, narot)`，使用 `scipy.ndimage.map_coordinates`，无批量接口。

**根因**：
- GPU 已有 `_rotate_volume_torch_gpu_batch`，CPU 仍为逐角度 for-loop。
- 典型配置下数百角度/颗粒，每角度一次 `map_coordinates`，单颗粒 CPU 耗时仍在 0.5–2 s 量级。

**方案**：
- **Numba 加速**：用 Numba `@njit` 重写 `rotate_volume` 内层（坐标变换 + 插值），与 SciPy 结果对比验证。
- **或粗粒度采样建议**：文档中建议探索阶段使用 cone_step=30、inplane_step=30，收敛后再细化。
- **预期**：Numba 实现可降 20–50% 单颗粒 CPU 时间（视实现质量）。

## 3. 解决方案优先级与预期收益

| 优先级 | 卡点 | 方案摘要 | 预期收益 |
|--------|------|----------|----------|
| **P0** | 2.1 GPU 网格 CPU 计算 | 网格计算迁移到 GPU | 减少 CPU-GPU 传输与等待，提升 GPU 利用率 |
| **P0** | 2.2 GPU NCC 逐角度 | 批量 FFT NCC | 每批从 B 次 FFT 降为 1 次，显著减少 kernel 启动 |
| **P1** | 2.3 Classification GPU swap 串行 | GPU 路径 ref 级 ThreadPoolExecutor 并行 | swap=True 且多 ref 时显著加速 |
| **P1** | 2.4 GPU 路径无预取 | 主进程预取队列 | I/O 瓶颈时吞吐提升 10–30% |
| **P2** | 2.5 多进程与 GPU 互斥 | 文档化；中期探索每 worker 单 GPU | 配置合理化 |
| **P2** | 2.6 CPU rotate_volume | Numba 加速或文档化粗粒度建议 | CPU 路径 20–50% 提升 |

## 4. 实施路线图（仅规划）

### 阶段一：高优先级（P0）

1. **GPU 网格计算迁移**
   - 在 `_rotate_volume_torch_gpu_batch` 内，用 PyTorch 在 device 上生成 `grid_batch`，替代 NumPy 循环。
   - 对 ZXZ 欧拉角，预计算旋转矩阵 `[B, 3, 3]`，用 `torch.einsum` 或 `torch.bmm` 与坐标张量相乘得到采样网格。

2. **批量 FFT NCC**
   - 新增 `_ncc_volume_fft_torch_batch(part_eval_t, ref_rot_batch)`，支持 `ref_rot_batch [B, D, H, W]`。
   - 在 `_align_single_scale_torch_gpu` 的 batch 分支中，用批量 NCC 替代 per-angle 循环，一次性得到 B 个 NCC 体积，再逐角度在 shift_search_mask 内 argmax。

### 阶段二：中优先级（P1）

3. **Classification GPU swap ref 并行**
   - 在 `_align_particle_task` 内，当 `swap and nref > 1` 且 `resolved_device == "cuda"` 时，用 `ThreadPoolExecutor(max_workers=min(nref, 8))` 并行 `align_one_particle` 到各 ref，主线程汇总 best_ref。

4. **GPU 路径预取**
   - 在 alignment/classification 的 GPU 分支（`_run_one` / `_align_particle_task` 的调用方）增加预取队列，后台线程预加载 2–4 个颗粒，主循环消费。

### 阶段三：低优先级（P2）

5. **多进程与 GPU 互斥**
   - 在 README/配置说明中明确「num_workers>1 时 classification 使用 CPU」，并给出选型建议。

6. **CPU rotate_volume**
   - 评估 Numba 重写 `rotate_volume` 的可行性，或补充粗粒度采样文档建议。
