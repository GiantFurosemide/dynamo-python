# pydynamo 性能卡点分析与解决方案规划（jg_016）

本文档基于 jg_012.md、jg_015.md 及对 pydynamo 代码库的实际浏览，针对**实际运行性能仍达不到使用需求**的问题，识别当前卡点并提出解决方案。**仅输出方案，不实现代码**。

---

## 1. 现状与已实现项回顾

| jg_012/jg_015 项 | 实现情况 | 备注 |
|------------------|----------|------|
| Alignment CPU 多进程 | ✓ | ProcessPoolExecutor + initializer 共享 ref/mask |
| Classification 流式累加、避免重复读盘 | ✓ | 对齐时即累加 per_ref_acc，平均阶段不重读 |
| FFT-based 平移 NCC | ✓ | mask 全 1 时使用 `_ncc_volume_fft` |
| Reconstruction 多进程 | ✓ | recon_workers + 按块流式处理 |
| 粒子 I/O 减少拷贝 | ✓ | np.asarray + 按需 copy |
| Crop 可配置 workers | ✓ | num_workers 控制 |

## 2. 新发现的性能卡点

### 2.1 卡点一：Alignment/Classification 每粒子 IPC 传输变换体积（tr）

**现象**：60k 颗粒时，主进程与 worker 间 IPC 成为显著瓶颈。

**根因**：

- 当前设计：每个 worker 任务返回 `(i, row, tr)`，其中 `tr` 为 `apply_inverse_transform` 后的 3D 体积。
- 单颗粒体积：48³×4B ≈ 0.44 MB，64³×4B ≈ 1 MB。
- 60k 颗粒 × 1 MB ≈ **60 GB** 需通过 pickle/队列在进程间传输。
- 主进程对每个结果执行 `avg_acc += tr`，串行累加，且反序列化成本高。

**量级估算**：若单次 IPC 传输 1 MB 耗时 ~1–5 ms，60k × 2 ms ≈ **2 分钟** 仅花在 IPC 上，且会随 worker 数增加而加剧（更多结果需回收）。

**方案**：

- **按 chunk 提交任务**：将 60k 颗粒打成 chunk（如每 chunk 50–200 颗粒），每个任务处理一整块。
- Worker 内：读颗粒 → 对齐 → 局部累加 `local_acc += tr` → 丢弃颗粒；只返回 `(rows_list, local_acc)`。
- 主进程：合并 `avg_acc += local_acc`，合并 rows。
- 预期：IPC 次数从 60k 降为 60k/chunk_size（如 300–1200 次），单次传输体积略增（local_acc 与 rows），但总传输量大幅下降（不再传输 60k 个独立 tr）。

---

### 2.2 卡点二：Classification 未使用 initializer 共享 ref/mask

**现象**：Classification 多进程时，每个 worker 首次任务需从磁盘加载所有 reference 与 mask。

**根因**：

- Alignment 已用 `_alignment_worker_init` + `initargs=(ref_vol, align_mask, wedge_mask, align_params)` 注入 worker。
- Classification 使用 `_classification_worker_cache`，在 worker 内按 `(config_path, ref_paths)` 缓存；**首次命中时**每个 worker 独立 `mrcfile.open` 加载 refs。
- N 个 worker → N 份 ref 内存；多 ref（如 4–8 个）时，内存与冷启动 I/O 成倍增加。

**方案**：

- 引入 `_classification_worker_init`，与 alignment 类似：主进程加载 `ref_vols`、`align_mask`、`wedge_mask`、`align_params`，通过 `ProcessPoolExecutor(initializer=..., initargs=...)` 注入。
- Worker 内使用全局变量，不再按 payload 中的 `ref_paths` 从磁盘加载。
- 预期：降内存、消除多 worker 重复读盘，缩短首批任务完成时间。

---

### 2.3 卡点三：带 nmask 时回退到实空间 NCC，单颗粒耗时激增

**现象**：用户使用 `nmask`（非全 1 mask）时，`_ncc_volume_fft` 返回 `None`，对齐回退到逐平移实空间 NCC。

**根因**：

- `core/align.py` 第 363–365 行：`if mask is not None and not np.all(mask): return None`。
- 实空间 NCC：每个角度 × 每个平移候选都要 `shift` + `_compute_cc_np`，计算量 O(角度数 × 平移数 × 体素数)。
- 典型配置下，单颗粒耗时可从 ~0.5 s（FFT NCC）增至 **数秒**（实空间 NCC）。

**方案**：

- **短期**：在文档中明确「nmask 会显著降低对齐速度」，建议在可接受精度下优先使用全 1 mask 或更大 mask 覆盖。
- **中期**：探索 mask 下的 FFT NCC 近似：
  - 方案 A：在 mask 内做 FFT NCC（mask 外置零），与 Dynamo 行为对比验证；
  - 方案 B：分块 FFT 或加权 FFT，在保持精度的前提下减少计算。
- 需与现有 `cc_mode`、`wedge` 等行为一致，并保留回归测试。

---

### 2.4 卡点四：GPU 路径单颗粒单角度循环，利用率低

**现象**：GPU 路径下，`_align_single_scale_torch_gpu` 对每个角度循环调用 `_rotate_volume_torch_gpu`、`_shift_tensor_zero`、`_ncc_torch`，kernel 启动与 CPU–GPU 往返频繁。

**根因**：

- 单颗粒、单角度的 for-loop，无法利用 GPU 批量计算。
- 每个角度一次 rotation + 多次 shift + NCC，kernel 启动开销占比高。

**方案**：

- **批量角度**：对同一颗粒，预计算多个角度的 ref_rot，堆叠为 batch，一次性在 GPU 上做 shift + NCC，取 argmax。
- **批量颗粒**：若内存允许，多颗粒 × 多角度组成 batch，进一步摊薄 kernel 启动成本。
- 需评估 PyTorch `grid_sample`、自定义 kernel 的 batch 扩展难度，以及多 GPU 下的负载均衡。

---

### 2.5 卡点五：Crop 同步写盘阻塞 worker

**现象**：Crop 时，每个 `_crop_one_with_volume` 内直接调用 `save_subtomo`，写盘与计算耦合。

**根因**：

- 60k 个小 MRC 文件写入，若存储为 HDD 或网络盘，小文件 I/O 成为瓶颈。
- Worker 线程在写盘时阻塞，无法继续 crop 下一颗粒。

**方案**：

- **异步写队列**：Worker 将 crop 结果（subtomo array + 路径）放入有界队列，专用写线程/进程负责 `save_subtomo`。
- 队列满时 worker 阻塞，避免内存无限增长。
- 预期：计算与 I/O 重叠，高 I/O 负载时提升吞吐。

---

### 2.6 卡点六：Crop 多 tomogram 时组内并行度受限

**现象**：当 tomogram 数量较多且每 tomogram 颗粒数较少时，`max_group_workers = min(num_workers, len(tomo_groups))` 导致部分 worker 处理「大组」，组内仅 1 线程。

**根因**：

- `commands/crop.py` 第 264–271 行：多 tomogram 时，`_process_tomo_group` 的 `num_workers` 被固定为 `1`。
- 即：每个 tomogram 组内串行处理颗粒，仅在不同 tomogram 组之间并行。

**方案**：

- 当 `len(tomo_groups) < num_workers` 时，对「大组」分配更多组内线程：例如 `group_workers = max(1, num_workers // len(tomo_groups))`。
- 当 `len(tomo_groups) >= num_workers` 时，保持当前逻辑（每组 1 线程，组间并行）。

---

### 2.7 卡点七：60k 量级下 ProcessPoolExecutor 调度与序列化开销

**现象**：60k 次 `submit` + 60k 次 `result()`，进程池队列与 pickle 开销不可忽视。

**根因**：

- 每任务 payload 含 `seed_row`（dict）、`full_path` 等；result 含 `row`（dict）、`tr`（ndarray）。
- 大量小任务导致调度器与序列化占比上升。

**方案**：

- 与 2.1 结合：**按 chunk 提交**，将 60k 打成 300–1200 个 chunk，每 chunk 处理 50–200 颗粒。
- Chunk payload：`(indices, paths_chunk, seed_rows_chunk)`。
- Chunk result：`(rows_list, local_acc)` 或 `(rows_list,)`（若不需要主进程累加 average，如 classification 的 per_ref 在 worker 内完成则需不同设计）。
- 注意：Classification 的 per_ref_acc 需在主进程合并，因此 chunk worker 应返回 `(rows_list, per_ref_local_acc_list)`，主进程合并。

---

### 2.8 卡点八：MRC 只读路径未充分利用 mmap

**现象**：`mrcfile.open(..., mode="r")` 在部分场景下可能返回 mmap-backed array，但当前 `np.asarray(mrc.data, dtype=np.float32)` 在非 writeable 时会 `copy()`。

**根因**：

- 多进程 fork 后，若子进程以只读方式访问同一 MRC，理论上可共享物理页（copy-on-write）。
- 若强制 copy，则每 worker 持有一份完整拷贝，内存与带宽压力大。

**方案**：

- 确认 `mrcfile` 在 `mode="r"` 下的行为；若已是 mmap，则避免不必要的 `copy()`。
- 在「只读 + 不修改」的路径上，统一使用 `np.asarray(..., copy=False)` 或等价方式，仅在必须 writeable 时 copy。
- 文档说明：多 worker 时，相同 MRC 路径可能被多进程打开，mmap 可降低总内存。

## 3. 解决方案优先级与预期收益

| 优先级 | 卡点 | 方案摘要 | 预期收益 |
|--------|------|----------|----------|
| **P0** | 2.1 每粒子 IPC 传输 tr | 按 chunk 提交，worker 内累加后返回 (rows, local_acc) | 60k 时 IPC 从 ~60 GB 降至 ~数百 MB，wall time 显著缩短 |
| **P0** | 2.2 Classification 无 ref 共享 | 增加 initializer 注入 ref_vols/mask/wedge | 降内存、减 I/O，多 ref 时收益明显 |
| **P1** | 2.3 带 mask 的 FFT NCC | 探索 mask 下 FFT NCC 近似或文档化降速说明 | 使用 nmask 时单颗粒时间可降一个数量级 |
| **P1** | 2.4 GPU 批量 kernel | 多角度/多颗粒 batch | GPU 利用率与吞吐提升 |
| **P1** | 2.7 60k 调度开销 | 与 P0 合并，chunk 提交 | 降低 submit/result 次数 |
| **P2** | 2.5 Crop 异步写 | 写队列 + 专用写线程 | I/O 瓶颈时提升 CPU 利用率 |
| **P2** | 2.6 Crop 组内并行 | 大组分配更多组内线程 | 多 tomo、少颗粒/组时加速 |
| **P2** | 2.8 MRC mmap | 只读路径避免不必要 copy | 降内存、略重叠 I/O |

## 4. 实施路线图（仅规划）

### 阶段一：高优先级（P0）

1. **Alignment chunk 提交**  
   - 新增 `_alignment_chunk_worker`：接收 `(indices, paths_chunk, seed_rows_chunk)`，返回 `(rows_list, local_acc)`。  
   - 主进程：按 chunk 提交，合并 rows 与 avg_acc。  
   - 配置项：`chunk_size`（默认 100，可调）。

2. **Classification initializer + chunk**  
   - 新增 `_classification_worker_init`，注入 ref_vols、align_mask、wedge_mask、params。  
   - 新增 `_classification_chunk_worker`：处理多颗粒，返回 `(rows_list, per_ref_local_acc_list)`。  
   - 主进程合并 per_ref_acc。

### 阶段二：中优先级（P1）

3. **带 mask 的 FFT NCC**  
   - 调研 Dynamo/RELION 在 mask 下的实现；实现或文档化。

4. **GPU 批量**  
   - 设计 batch 接口，先实现「单颗粒 × 多角度」batch，再扩展「多颗粒 × 多角度」。

### 阶段三：低优先级（P2）

5. **Crop 异步写、组内并行**  
   - 实现写队列与组内 worker 分配逻辑。

6. **MRC mmap 优化**  
   - 审计只读路径，统一 copy 策略。
