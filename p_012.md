# p_012: pydynamo 性能优化 — 细化实现方案与测试方案

基于架构规划 [jg_012.md](jg_012.md)，本文档给出细化实现方案与测试方案，供开发与测试执行。

---

## 1. 实现方案（按优先级）

### 1.1 P0：Alignment / Classification CPU 多进程并行

**目标**：CPU 路径下使用多进程处理粒子，利用多核。

**配置**：
- 在 alignment / classification 的 YAML 中增加 `num_workers`（与 crop 命名一致）。
- 默认值：`1`（保持现有单进程行为）；`<=0` 表示自动：`max(1, os.cpu_count() - 1)` 或 `os.cpu_count()`，由实现决定。
- 仅当 `device == "cpu"`（或等效）且 `num_workers > 1` 时启用 ProcessPoolExecutor。

**Alignment**（`pydynamo/src/pydynamo/commands/alignment.py`）：
- 在 `run()` 中读取 `num_workers`，解析方式可复用 crop 的 `_resolve_num_workers` 逻辑（或在本模块内实现 `_resolve_cpu_workers`）。
- 当前逻辑：`resolved_device == "cuda" and len(gpu_ids) > 1` 时用 `ThreadPoolExecutor`；否则为单线程 for 循环（约 289–318 行）。
- 新增分支：当 `resolved_device == "cpu" and num_workers > 1` 时，使用 `concurrent.futures.ProcessPoolExecutor(max_workers=num_workers)`。
- Worker 任务：每个 task 为 `(i, p_path, full_path)`；worker 内需能加载 reference、mask、wedge 等。为避免大对象序列化，采用「worker 内按 config 路径加载」：主进程只传 (config_path, task_tuple) 或 (reference_path, mask_path, …) + task_tuple，worker 内打开 MRC/加载 mask 一次；或主进程加载 ref/mask，通过 fork 继承（仅 Linux/macOS），子进程只读。**推荐**：主进程加载 ref_vol、align_mask、wedge_mask 等，通过 `initializer` 将路径或共享方式传入，或每个 worker 根据传入的路径自行加载（避免传大数组）。
- 实现要点：ProcessPoolExecutor 在 spawn 下无法继承主进程内存，因此要么用 fork 上下文（Linux），要么 worker 内根据路径加载 ref/mask。为兼容 Windows，采用 **worker 内按路径加载**：主进程将 `reference`、`mask_path`、wedge 参数、以及表行种子等通过可序列化结构传入；每个 worker 首次执行时加载 ref 与 mask（可做进程内缓存），再执行「读粒子 MRC + align_one_particle + apply_inverse_transform」。
- 主进程：收集 `(i, row, tr)`，排序后合并 rows 与 avg_acc/avg_used；写表与 average 逻辑不变。

**Classification**（`pydynamo/src/pydynamo/commands/classification.py`）：
- 同样增加 `num_workers` 解析；当 `resolved_device == "cpu" and num_workers > 1` 时使用 ProcessPoolExecutor。
- 每轮迭代内，对齐阶段：任务为 (pidx, p_path) → worker 读 MRC、对多个 ref 做 align、返回 (pidx, best_ref, best_row)；主进程写 JSONL 行到各 ref 的 row_fh，并记录 per_ref_counts。
- Worker 内需能访问多个 ref_vol、align_mask、wedge 等；同样采用「worker 内按路径加载」或传序列化参数，避免主进程传大数组。

**共享工具**：
- 可在 `runtime` 或 `commands` 下提供 `_resolve_cpu_workers(config_value, default=1)`：`<=0` 时返回 `max(1, (os.cpu_count() or 1) - 1)`，否则返回 `max(1, int(config_value))`。
- 配置文件：`pydynamo/config/alignment_defaults.yaml`、`pydynamo/config/classification_defaults.yaml` 增加 `num_workers: 1` 及注释。

### 1.2 P0：Classification 每轮迭代粒子只读一次

**目标**：同一轮迭代内，每个粒子文件只打开一次。

**方案**（二选一，推荐 A）：
- **方案 A（流式对齐 + 按 ref 累积）**：在对齐阶段，每完成一个粒子即得到 (best_ref, pose)。在该粒子内存尚未释放时，立即 `apply_inverse_transform` 并累加到对应 ref 的 buffer；然后丢弃粒子内存。平均阶段不再按粒子路径重新读 MRC，只对各 ref buffer 做归一化、对称性等后处理，写 average MRC。这样每粒子整轮只读 1 次。
- **方案 B（平均阶段按 path 去重）**：对齐阶段只输出 (particle_path, ref_id, pose) 到 JSONL。平均阶段：收集所有 ref 的 JSONL，按 `rlnImageName` 去重得到唯一 path 列表；对每个 path 只读一次 MRC，根据 (path → list of (ref_id, pose)) 将变换结果累加到对应 ref buffer。实现稍复杂（同一粒子可能只属于一个 ref，但数据结构上要支持 path→ref_id 映射）。

**实现（方案 A）**（`pydynamo/src/pydynamo/commands/classification.py`）：
- 在每轮迭代开始时，为每个 ref 分配 `acc = np.zeros((sidelength, sidelength, sidelength), dtype=np.float64)` 和 `used = 0`。
- 对齐循环（单线程或多进程收集结果后）：每得到 (pidx, best_ref, best_row)，若 best_row 非空，则在该粒子数据可用处（当前实现是主进程在收到 worker 结果时没有粒子数据，因此需在 worker 内同时返回 **变换后的体积 tr**，或主进程在收到 best_row 后再读一次粒子——后者仍会重复读）。因此方案 A 若要「只读一次」，必须在 **对齐的 worker 内** 完成「读 MRC → align → 得到 best_ref/pose → apply_inverse_transform(particle, pose) → 返回 (pidx, best_ref, best_row, tr)」；主进程只做 `acc[best_ref] += tr`，不再读盘。
- 即：对齐阶段每个 task 返回 `(pidx, best_ref, best_row, tr)`，其中 `tr` 为已变换到标准朝向的体；主进程按 best_ref 累加；平均阶段仅 `avg = acc / used` 等后处理，不再 open(particle_path)。

**实现（方案 B）**：平均阶段先扫描所有 ref 的 row 文件，收集 `set(rlnImageName)`，然后对每个 path 只 open 一次，查 (path → ref_id, pose)，累加到对应 ref。需要先写 row 再读 row 做平均，顺序不变即可。

**推荐实现**：方案 A，且对齐 worker 返回 `(pidx, best_ref, best_row, tr)`；若为多进程，worker 内读 MRC、对齐、得到 best_ref/pose，再 apply_inverse_transform 得到 tr，返回；主进程累加 tr 到 per-ref buffer。单线程路径同样：每粒子对齐后立即 tr = apply_inverse_transform(...)，累加到 ref buffer，再写 row。

### 1.3 P1：Alignment 平移搜索 FFT-based NCC

**目标**：对给定角度，用 FFT 一次性计算 3D 平移 NCC 曲面，再取峰值与亚像元细化，减少逐平移 shift+NCC 的计算量。

**文件**：`pydynamo/src/pydynamo/core/align.py`。

**要点**：
- 在 `_align_single_scale`（或等价单尺度函数）中，对每个角度 (tdrot, tilt, narot)：
  - 将 reference 旋转到该角度并做必要预处理（mask、wedge 等）。
  - 与 particle 做 **归一化互相关** 的 FFT 实现：NCC(u,v,w) = IFFT( FFT(part) * conj(FFT(ref_rot)) ) / (norm 归一化)；或使用现有 `normalized_cross_correlation` 的 FFT 版本。
- 得到 3D NCC 体积后，在整数 shift 搜索范围内取最大值位置，再在该位置做亚像元细化（现有 `_subpixel_offset_3d_quadratic` 等）。
- 保持与现有 API 一致：仍支持 wedge、mask、subpixel、shift_mode（cube/ellipsoid 等）。shift 搜索范围由 `_iter_integer_shifts` 定义，FFT 给出全体积 NCC，只需在对应位置取样或取局部最大值。
- 需与 realspace NCC 结果在数值上接近（可加回归测试：同一 particle/ref，比较 FFT 路径与 realspace 路径的 peak 位置与 CC 值是否一致或误差在容忍范围内）。

### 1.4 P1：Reconstruction 多进程按块累加

**目标**：Reconstruction 阶段使用多进程按粒子块累加，主进程合并。

**配置**：在 reconstruction 的 YAML 中增加 `recon_workers`，默认 `1`；`>1` 时启用 ProcessPoolExecutor。

**文件**：`pydynamo/src/pydynamo/commands/reconstruction.py`。

**实现**：
- 将 `paths_list` 与对应的 angles/shifts 分成若干 chunk（每块若干粒子）。
- 每个 chunk 任务：worker 内对块内粒子逐个 open MRC、apply_inverse_transform、累加（realspace 路径为实空间 sum；Fourier 路径为 fft_sum + wedge_mask 逻辑）；返回 (partial_sum, count) 或 (fft_sum, count)。
- 主进程：启动 ProcessPoolExecutor，submit 各 chunk；收集 (partial_sum, count)，合并 sum 与 count，最后求平均、写 MRC。Fourier 路径合并 fft_sum 与 denom。
- Worker 内同样按路径打开 MRC，避免主进程传大数组；可传 base_dir、subtomograms、paths 子列表、angles 子数组等。

### 1.5 P2：粒子 I/O 使用只读视图（mmap）

**目标**：在不需要修改粒子数据时，使用只读视图代替 `.data.copy()`，降低内存与带宽。

**实现**：
- 在所有「仅读取粒子并做对齐/变换/累加」的路径中，将 `mrc.data.copy()` 或 `.astype(np.float32)` 改为：先 `vol = np.asarray(mrc.data, dtype=np.float32)` 或保持只读视图；若后续有 in-place 操作（如 `* mask`），则在局部做 `vol = np.asarray(vol, dtype=np.float32).copy()` 再操作，或先 copy 再传下游。
- 检查 alignment、classification、reconstruction 中所有 `mrcfile.open(...).data.copy()`：若该分支仅读且不会写回文件，则改为只读视图 + 在需要写时再 copy。注意 mrcfile 的 `data` 可能是 memmap；numpy 操作若会写视图可能影响，故在「需要修改」时再 copy。
- 文档与注释中说明：只读路径使用视图，多进程 fork 后可共享只读页。

## 2. 测试方案

### 2.1 单元测试

- **num_workers / recon_workers 解析**：
  - 新增或扩展现有测试：`_resolve_cpu_workers(0)`、`_resolve_cpu_workers(-1)` 返回 >=1；`_resolve_cpu_workers(1)` 返回 1；`_resolve_cpu_workers(4)` 返回 4。
  - Reconstruction：`recon_workers` 解析为 1 或指定值。
- **对齐/分类**：Mock 或小数据下，`num_workers=1` 与 `num_workers=2` 结果一致（表行顺序、CC 值、average 数值接近）。

### 2.2 集成测试

- **Alignment**：使用 synthetic 小数据集，`device=cpu`，`num_workers=1` 跑一次，保存 table 与 average；再 `num_workers=2` 跑一次；对比两次数值（允许浮点误差）。
- **Classification**：单轮或两轮，对比「流式累加」前后平均 ref 与表是否一致；或对比单线程与 2 worker 结果。
- **Reconstruction**：同一输入表与粒子，`recon_workers=1` 与 `recon_workers=2` 各跑一次，对比输出 MRC 数值接近。

### 2.3 Benchmark 与回归

- 扩展 `pydynamo/src/pydynamo/scripts/benchmark_profiles.py`：增加 profile「多进程 alignment」（如 num_workers=2,4）、「多 worker reconstruction」；输出吞吐或耗时到 JSON/日志。
- 现有 pytest：`test_alignment_command.py`、`test_classification.py`、`test_reconstruction_command.py` 等必须全部通过；新增用例覆盖 num_workers>1 与 recon_workers>1（若可行用 small synthetic 数据）。

### 2.4 回归与兼容性

- 默认 `num_workers=1`、`recon_workers=1` 时，行为与现有一致。
- FFT-based NCC 与 realspace NCC 的数值差异在测试中限定在可接受范围（如 CC 差异 < 1e-5，shift 差异 <= 1 voxel）。

## 3. 依赖与风险

- **多进程**：Windows 使用 spawn，无法 fork；ref/mask 不能依赖 fork 继承，必须 worker 内按路径加载或通过可序列化参数传递。
- **兼容性**：默认保持单进程，避免影响现有用户与 CI。
- **GIL**：CPU 密集型粒子级任务用 ProcessPoolExecutor，不用 ThreadPoolExecutor，以真正利用多核。
