# p_015: jg_015 60k 颗粒规模优化 — 细化实现方案与测试方案

基于架构规划 [jg_015.md](jg_015.md)，本文档给出可执行的实现方案与测试方案，并按 jg_015 §6 的优先级实施。

---

## 1. 目标与范围

- **目标**：在 60,000 颗粒规模下进一步降低内存与 I/O、明确推荐配置、增强可观测性。
- **不实现**：带 mask 的 FFT NCC、GPU 批量 kernel、Crop 异步写、I/O 预取（列为后续或低优先级）。

## 2. 实现方案（按优先级）

### 2.1 文档与推荐配置（高 — jg_015 §6.1）

**文件**：`pydynamo/README.md`（或 `pydynamo/docs/` 若存在）

**内容**：
- 新增小节「Large-scale runs (e.g. 60k particles)」或「Recommended settings for ~60k particles」。
- 插入 jg_015 §4 的推荐配置表（alignment num_workers 8–16、recon_workers 8–16、classification num_workers 8–16、crop num_workers 4–16；multigrid_levels=2；cone_step/inplane_step=15；max_iterations 3–5）。
- 补充说明：`num_workers=0`（auto）在 60k 时建议显式设为 8–16 以免占满机器；`recon_workers` 过大增加调度与合并成本，推荐 8–16；crop 的 `num_workers` 建议不超过 tomogram 数 × 每 tomo 期望并行数或 CPU 核数。

**验收**：README 中可查 60k 量级推荐参数表及简短说明。

### 2.2 Alignment CPU 多进程：Ref/Mask 共享（高 — jg_015 §3.1）

**问题**：当前 `_alignment_cpu_worker` 每任务内从磁盘加载 reference 与 mask，60k 任务即 60k 次加载；多进程下同一 worker 进程会重复加载 ref/mask。

**方案**：使用 ProcessPoolExecutor 的 **initializer**，在主进程加载 ref_vol、align_mask、wedge_mask 一次，与对齐参数字典一起通过 `initargs` 传入，在 worker 进程内设为模块级全局；worker 函数仅接收 (i, p_path, full_path, seed_row)，不再在任务内读 ref/mask。

**文件**：`pydynamo/src/pydynamo/commands/alignment.py`

**实现要点**：
- 新增 `_alignment_worker_init(ref_vol, align_mask, wedge_mask, align_params)`：将 ref_vol、align_mask、wedge_mask、align_params 存入模块级全局（如 `_worker_ref`, `_worker_mask`, `_worker_wedge`, `_worker_params`）；`align_params` 为 dict，包含 cone_step、tdrot_step、…、wedge_apply_to、fsampling_mode、subpixel_method 等 align_one_particle 所需标量/字符串，不含大数组。
- 新增 `_alignment_cpu_worker_impl(payload)`：payload 仅含 i, p_path, full_path, seed_row；从全局读取 ref/mask/wedge/params；加载粒子 MRC；调用 align_one_particle；返回 (i, row, tr)。原 `_alignment_cpu_worker` 保留为「无 initializer 时」的兼容路径，或删除并由主路径统一使用 initializer。
- 主进程：在「CPU 且 num_workers > 1」分支中，先构建 ref_vol、align_mask、wedge_mask（与现有一致），再构建 align_params dict；创建 `ProcessPoolExecutor(max_workers=num_workers, initializer=_alignment_worker_init, initargs=(ref_vol, align_mask, wedge_mask, align_params))`；submit 的 payload 仅 (i, p_path, full_path, seed_row)。
- 注意：Windows 使用 spawn 时 ref_vol/align_mask/wedge_mask 会序列化到各 worker，每个 worker 一份，总内存 ≈ N_workers × (ref + mask + wedge)，可接受；Linux/macOS fork 时可由子进程继承只读内存，initializer 中仅覆盖全局即可。

**验收**：多进程 alignment 时每个 worker 进程仅加载一次 ref/mask；`pytest test/test_alignment_command.py` 通过；与 num_workers=1 结果一致（或差异在数值容忍范围内）。

### 2.3 Reconstruction chunk 内流式处理说明（高 — jg_015 §3.2）

**现状**：`_reconstruction_chunk_worker` 已按「逐颗粒：open MRC → transform → 累加 → 丢弃」实现，未在内存中保留整块颗粒。

**方案**：在 `_reconstruction_chunk_worker` 的 docstring 或循环上方增加注释，明确「chunk 内流式：读一个、算一个、累加、丢弃，不保留整块颗粒」，便于审查与后续维护；并在 README 或 r_015 中注明「reconstruction 已保证 chunk 内流式」。

**文件**：`pydynamo/src/pydynamo/commands/reconstruction.py`

**验收**：注释存在且与实现一致。

### 2.4 Benchmark 60k 量级 profile（中 — jg_015 §6.2）

**文件**：`pydynamo/src/pydynamo/scripts/benchmark_profiles.py`

**内容**：
- 新增 profile：如 `60k_mp4`（particles=1000，box=48，num_workers=4）、`60k_mp8`（particles=1000，box=48，num_workers=8），用于近似 60k 规模下的多进程 alignment 吞吐与回归；或单一 profile `large_mp8`（particles=1000，box=48 或 64，num_workers=8）。

**验收**：`--profiles 60k_mp4,60k_mp8`（或所选名称）可运行且输出 JSON；与现有 small/medium 格式一致。
